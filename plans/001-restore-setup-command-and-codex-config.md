## Why this matters

New Conductor workspaces currently fail during setup because shared Conductor
settings call a script name that no longer exists. The same branch also has a
tracked `.codex/config.toml` that points at a wrapper script that is absent from
the workspace. Until setup is compatible and generated config is reproducible,
the intended single `mcut-live` development server cannot be relied on by
agents or humans.

## Current state

- `package.json` owns root scripts. At `package.json:12-16` it currently defines `dev`, `dev:web`, `setup`, and `mcut:dev`, but no `codex:setup:mcut`.
- `.conductor/settings.toml` owns shared Conductor workspace setup. At `.conductor/settings.toml:5` it currently runs `bun install --frozen-lockfile && bun run codex:setup:mcut`.
- Running `bun run codex:setup:mcut` currently fails with `error: Script not found "codex:setup:mcut"`.
- `scripts/setup-codex-mcut-env.ts:70-81` writes a Codex MCP config that uses `scripts/codex-mcp.sh` through `git rev-parse --show-toplevel`.
- `.codex/config.toml:3-6` currently points at an absolute `scripts/codex-mcp-live.sh` path, but `scripts/codex-mcp-live.sh` is not present.
- Repo convention: Conductor shared setup belongs in `.conductor/settings.toml`; repo-local Codex setup is generated by `scripts/setup-codex-mcut-env.ts`.

## Steps

### Step 1: Restore backward-compatible setup entrypoints

In `package.json`, keep the canonical `setup` script and add back a
`codex:setup:mcut` alias that runs the same `scripts/setup-codex-mcut-env.ts`.
Then update `.conductor/settings.toml` to use the canonical script:

```toml
setup = "bun install --frozen-lockfile && bun run setup"
```

The alias is still required because older docs, existing workspaces, and user
muscle memory may call `bun run codex:setup:mcut`.

**Verify**: `bun run codex:setup:mcut` -> exits 0 and prints `Configured worktree-local Codex environment for mcut.`

### Step 2: Regenerate and commit the tracked Codex config shape

Run `bun run setup`, then inspect `.codex/config.toml`. It should register only
`mcut-live`, should call `scripts/codex-mcp.sh`, and should not contain
`scripts/codex-mcp-live.sh` or an absolute worktree `cwd`.

Expected shape:

```toml
approval_policy = "never"

[mcp_servers.mcut-live]
command = "bash"
args = ["-lc", 'exec "$(git rev-parse --show-toplevel)/scripts/codex-mcp.sh"']
startup_timeout_sec = 20
tool_timeout_sec = 300
default_tools_approval_mode = "approve"
```

**Verify**:

```sh
bun run setup
rg -n 'codex-mcp-live|cwd =|mcp_servers\.mcut\]' .codex/config.toml
```

The `rg` command should return no matches. Then run:

```sh
rg -n 'scripts/codex-mcp\.sh|mcp_servers\.mcut-live' .codex/config.toml
```

Expected: both patterns are present.

### Step 3: Keep the wrapper executable and repo-root-relative

Confirm `scripts/codex-mcp.sh` exists and resolves the repo root from its own
location before running the local source MCP bridge:

```sh
ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
cd "$ROOT"
exec bun packages/mcp-server/src/bridge-cli.ts mcp --port 44737 "$@"
```

If the script is not executable, make it executable.

**Verify**: `test -x scripts/codex-mcp.sh && echo ok` -> prints `ok`.

### Step 4: Run focused tests

Run the focused tests that cover the live MCP server and Studio MCP surface.

**Verify**:

```sh
bun test packages/mcp-server/src/server.test.ts
bun run --filter=mcut-studio-web test
```

Expected: both commands exit 0.
